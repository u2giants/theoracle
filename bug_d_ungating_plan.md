# Bug D Ungating Plan

## Problem Restatement

Bug D is the failure mode where the Oracle can extract many quote-backed atomic claims from a workflow diagram but still cannot expose the holistic workflow layer until a large pending claim queue is manually approved.

The original diagnosis says generated macro relationships were born blocked_pending_support whenever any support claim was not already approved, while the canonical Pop Creations swimlane run had 235 of 241 claims in pending_review ([fix_enhancement.md:193](fix_enhancement.md:193), [fix_enhancement.md:202](fix_enhancement.md:202)). The same section ties the failure to weak dedup: 241 claims came from 56 unique quotes because the same diagram edge could be phrased as dependency, process_rule, or policy and hash differently ([fix_enhancement.md:146](fix_enhancement.md:146), [fix_enhancement.md:726](fix_enhancement.md:726), [fix_enhancement.md:730](fix_enhancement.md:730)).

Current source has already moved partway toward the desired design: workflow maps exist, document ingestion records workflowTrace, computeCandidateHash accepts stableGraphKey, generated macro relationships now enter pending_review, and read-time serving still checks support approval. This plan is therefore a hardening/backfill/verification plan, not a proposal to re-implement from scratch.

## Proposed Design

### 1. Edge-Level Dedup

Use a stable workflow-edge dedup key as a layer above the current summary-text hash. The canonical key should be derived from the workflow map, not model prose:

    workflow-map:{workflowMapId}:edge:{normalizedFromNode}:{normalizedToNode}:condition:{normalizedConditionOrNone}:owners:{normalizedFromOwner}->{normalizedToOwner}:facet:{assertionFacet}

Where possible, use durable workflow node IDs and edge IDs from source_workflow_maps.edges_json; include normalized labels/owners/condition as a defensive fallback for regenerated maps where edge IDs change. The business inputs match the diagnosis: source node, target node, condition label, and swimlane/owner(s) ([fix_enhancement.md:740](fix_enhancement.md:740), [fix_enhancement.md:746](fix_enhancement.md:746)).

Integration with current code:

- Keep computeCandidateHash as the central dedup primitive. It already replaces the summary component with graph:canonicalized stableGraphKey when stableGraphKey is present ([packages/oracle-engines/src/extraction/candidate-hash.ts:51](packages/oracle-engines/src/extraction/candidate-hash.ts:51), [packages/oracle-engines/src/extraction/candidate-hash.ts:53](packages/oracle-engines/src/extraction/candidate-hash.ts:53)).
- Harden workflowEdgeDedupeKey and matchWorkflowEdgeForQuote in document ingestion so the key is populated for broad extraction and lens extraction, not only the current broad path. Current broad ingestion already sets workflowTrace and passes stableGraphKey ([apps/workers/src/trigger/document-ingestion.ts:1038](apps/workers/src/trigger/document-ingestion.ts:1038), [apps/workers/src/trigger/document-ingestion.ts:1275](apps/workers/src/trigger/document-ingestion.ts:1275)); verify whether document-lens-extraction needs the same workflow trace path for its candidates.
- Promotion should remain unchanged conceptually: same hash means executePromotion serializes by advisory lock and appends validated evidence to the existing claim rather than inserting another claim ([packages/oracle-engines/src/extraction/promotion-executor.ts:337](packages/oracle-engines/src/extraction/promotion-executor.ts:337), [packages/oracle-engines/src/extraction/promote-candidate.ts:14](packages/oracle-engines/src/extraction/promote-candidate.ts:14)).
- Allow multiple claims per edge only when the assertion dimension is materially different. Do this by adding a small assertionFacet into the stable graph key when the candidate is not merely another wording of the edge transition. Suggested facets: transition, owner, system_entry, approval_gate, exception_condition, metric_or_sla. Do not include claimType or claimKind alone, because those are exactly the weak labels that caused duplicate dependency/process_rule/policy claims.
- Store enough trace metadata to audit dedup decisions. Current candidates already keep rawCandidateJson.workflowTrace ([apps/workers/src/trigger/document-ingestion.ts:1078](apps/workers/src/trigger/document-ingestion.ts:1078)). If additional schema is needed, prefer candidate/claim metadata columns or JSON fields over changing evidence semantics; evidence must remain the validated quote/chunk.

Acceptance target: on the Pop flow rerun, duplicate claim rate should be measured per workflowMapId + workflowEdgeId + assertionFacet, and the approval queue should collapse to one canonical transition claim per edge/facet plus genuinely distinct owner/system/condition claims.

### 2. Visibility Policy for blocked_pending_support

Separate review visibility from serving eligibility.

Admin visibility:

- Generated macro relationships with at least two support claims and no rejected/superseded support should be visible in Admin immediately. Current source implements this by returning pending_review from statusForGeneratedMacroRelationship ([packages/oracle-engines/src/macro/lifecycle.ts:16](packages/oracle-engines/src/macro/lifecycle.ts:16), [packages/oracle-engines/src/macro/lifecycle.ts:25](packages/oracle-engines/src/macro/lifecycle.ts:25)).
- Keep blocked_pending_support as a valid status for legacy rows or explicit workflows that should signal reviewable structure, but support not yet approvable. Admin already includes that status in ordering ([apps/web/app/admin/macro/page.tsx:48](apps/web/app/admin/macro/page.tsx:48), [apps/web/app/admin/macro/page.tsx:51](apps/web/app/admin/macro/page.tsx:51)).
- Add an Admin filter/view label for support approval needed that includes relationships whose support contains pending claims, regardless of whether the relationship row status is pending_review or blocked_pending_support. This avoids hiding the queue behind overloaded status names.

Serving policy:

- Never serve a relationship to Brain/chat/MCP unless macro_relationships.status = approved and every linked support claim is currently claims.status = approved. This is already enforced in getApprovedMacroRelationships by both the row status filter and read-time NOT EXISTS support check ([packages/oracle-engines/src/macro/approved-relationships.ts:63](packages/oracle-engines/src/macro/approved-relationships.ts:63), [packages/oracle-engines/src/macro/approved-relationships.ts:67](packages/oracle-engines/src/macro/approved-relationships.ts:67)).
- Keep approval impossible while support is pending. The UI disables the approve button unless all support is approved, and the server action throws if any support is not approved ([apps/web/app/admin/macro/page.tsx:195](apps/web/app/admin/macro/page.tsx:195), [apps/web/app/admin/macro/_actions.ts:54](apps/web/app/admin/macro/_actions.ts:54)).
- Preserve staleness sweeps for regressions: approved relationships whose support later leaves approved become stale_support ([packages/oracle-engines/src/macro/approved-relationships.ts:165](packages/oracle-engines/src/macro/approved-relationships.ts:165), [packages/oracle-engines/src/macro/approved-relationships.ts:171](packages/oracle-engines/src/macro/approved-relationships.ts:171)).

This preserves the trust contract because Admin can review draft structure early, but no employee-facing or synthesis-facing helper treats that structure as approved knowledge until the support claims are approved at read time.

### 3. Triage and Prioritization

The admin should not have to approve hundreds of near-duplicate claims before seeing the workflow backbone.

Design:

- Introduce a Backbone claims admin slice for pending claims that support high-value macro relationships. Compute it from macro_relationship_claims joined to macro_relationships and claims: pending claims first, ordered by relationship triage_score, relationship impact_score, relationship type priority, and support role.
- Relationship types from deterministic workflow paths should rank high: exception_path already gets triage 88 and sequence gets 78 in deterministic insertion ([apps/workers/src/trigger/macro-relationship-extraction.ts:444](apps/workers/src/trigger/macro-relationship-extraction.ts:444), [apps/workers/src/trigger/macro-relationship-extraction.ts:480](apps/workers/src/trigger/macro-relationship-extraction.ts:480)). Branch/exception coverage findings also already get higher triage than ordinary edges ([apps/workers/src/trigger/source-coverage-audit.ts:148](apps/workers/src/trigger/source-coverage-audit.ts:148), [apps/workers/src/trigger/source-coverage-audit.ts:149](apps/workers/src/trigger/source-coverage-audit.ts:149)).
- Add per-claim unlocks N macro relationships and highest triage relationship indicators in the claims review queue. Source of truth is the support table, not a denormalized claim column.
- In Admin Macro, show pending support claims as actionable links into claim review. The page already renders support summaries and statuses ([apps/web/app/admin/macro/page.tsx:209](apps/web/app/admin/macro/page.tsx:209), [apps/web/app/admin/macro/page.tsx:213](apps/web/app/admin/macro/page.tsx:213)); the next step is making that a focused approval workflow.
- Treat canonical edge claims as the review unit. If multiple pending claims share the same workflowTrace.workflowDedupeKey/facet, display one canonical candidate with attached duplicate evidence rather than separate rows.

Backbone query shape, described only: select pending claims linked through macro_relationship_claims to relationships in pending_review/blocked_pending_support/needs_review, group by claim, compute max relationship triage and count of relationships unblocked by approval, and sort descending. This can be a server-side admin query; no new public RLS path.

### 5. Step-by-Step Implementation Order

Small PRs, not one giant PR:

1. PR 1: Verification harness and baseline metrics. Add/extend a Pop-style workflow fixture and offline smoke that reports workflow map rows, edge trace coverage, duplicate rate per edge/facet, deterministic macro rows, pending support counts, and serving count. This builds on the existing fixture assertions for stableGraphKey and pending-review macro visibility ([packages/ai/src/__verify__/workflow-map-fixture.ts:99](packages/ai/src/__verify__/workflow-map-fixture.ts:99), [packages/ai/src/__verify__/workflow-map-fixture.ts:115](packages/ai/src/__verify__/workflow-map-fixture.ts:115)).
2. PR 2: Harden edge dedup key construction. Normalize from/to node IDs, labels, conditions, and owners; introduce assertionFacet; add unit tests proving paraphrased transition claims collide while owner/system/approval facts can remain distinct. Keep all behavior funneled through computeCandidateHash.
3. PR 3: Apply workflow trace/dedup consistently across extraction paths. Verify broad document ingestion and lens extraction both attach workflow trace where possible. Broad ingestion already does; lens extraction should be audited and updated only if missing. Add tests or a dry-run fixture proving lens fanout cannot recreate four duplicate claims for one edge.
4. PR 4: Admin backbone triage. Add a server-rendered Admin slice/query that surfaces pending canonical edge claims linked to high-triage relationships. Include unlocks N relationships and direct claim-review actions/links. No new public client access to macro tables.
5. PR 5: Legacy/backfill cleanup. For existing documents, compute candidate/claim edge groups from raw_candidate_json.workflowTrace, mark or collapse duplicates where safe, and requeue/recompute deterministic macro relationships. Do this as a guarded script or worker task, dry-run first, especially for the canonical document.
6. PR 6: Visibility/status hardening. Decide whether legacy blocked_pending_support rows should be migrated to pending_review for Admin visibility or kept as-is with a derived support pending filter. Add regression tests that pending-support rows appear in Admin but never appear from getApprovedMacroRelationships.
7. PR 7: Production verification. Rerun the canonical Pop Creations document, capture before/after metrics, and update fix_enhancement.md/handoff docs with actual counts: unique workflow edges, canonical edge claims, pending approval queue size, macro rows visible in Admin, approved/served macro rows after approving the backbone.

### 6. Risks and Invariants

Do not break these invariants:

- Atomic claims keep exact-quote validation against one chunk. Workflow maps, outlines, and graph prose never become quotable evidence ([fix_enhancement.md:464](fix_enhancement.md:464), [fix_enhancement.md:466](fix_enhancement.md:466), [DECISIONS.md:544](DECISIONS.md:544)).
- Approved macro relationships must verify every support claim is approved at read time, not merely trust macro_relationships.status or claim_status_at_link ([fix_enhancement.md:467](fix_enhancement.md:467), [DECISIONS.md:545](DECISIONS.md:545), [packages/oracle-engines/src/macro/approved-relationships.ts:67](packages/oracle-engines/src/macro/approved-relationships.ts:67)).
- Outline/workflow-map prose is guidance only. It can create structure, triage, and coverage expectations; it cannot validate a claim or make a relationship serveable by itself ([fix_enhancement.md:420](fix_enhancement.md:420), [fix_enhancement.md:424](fix_enhancement.md:424), [fix_enhancement.md:274](fix_enhancement.md:274)).
- Dedup must not collapse materially different facts. The stable edge key should collapse duplicate phrasings of the same transition, but the facet design must preserve distinct owner, system-entry, approval-gate, exception-condition, and metric/SLA assertions.
- Admin visibility must not become serving visibility. pending_review or blocked_pending_support relationships may be inspectable by admins, but Brain/chat/MCP must remain restricted to approved relationships with currently approved support.
- RLS/server boundary must hold. Macro/workflow tables stay service-role/server-only unless explicit RLS policies are added first ([fix_enhancement.md:468](fix_enhancement.md:468), [docs/architecture.md:903](docs/architecture.md:903)).
- Do not add a new Trigger.dev schedule; the repo is at its schedule limit and macro work should reuse existing on-demand/followup paths ([DECISIONS.md:546](DECISIONS.md:546), [fix_enhancement.md:462](fix_enhancement.md:462)).

## Concrete Acceptance Checks

- Pop flow rerun produces non-empty source_workflow_maps, non-empty raw_candidate_json.workflowTrace for diagram-edge claims, and deterministic macro relationships from workflow paths.
- Duplicate approval queue shrinks materially: duplicate rate is reported per workflow edge/facet, not only by summary text.
- Admin Macro shows pending-support relationships and their support claims ordered by triage; admin can identify the small backbone set to approve first.
- getApprovedMacroRelationships returns zero for pending-support relationships before support approval, then returns approved relationships only after every support claim is approved.
- No claim or macro relationship uses outline/workflow-map prose as evidence; all atomic evidence remains validated document_chunk quotes.
